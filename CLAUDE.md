# Personal Trainer — Context Graph Demo

Technical interview demo built to prove understanding of a **context graph** for a solo fitness coaching app.

## What this is

A working demo — not a production app — built to show how an AI agent can make decisions grounded in a timestamped graph rather than relying on an LLM to decide. The chatbot responses are scripted/fallback for demo purposes (see note below), but the graph traversal and rule engine are real.

## Stack

**Backend** — `backend/`
- Python 3.10+ with [uv](https://github.com/astral-sh/uv) (package manager)
- FastAPI — pure JSON API, no HTML templating
- networkx — in-memory directed graph (the context graph)
- Anthropic SDK (`claude-sonnet-4-6`) — language generation only, never decisions
- pydantic v2 — request/response schemas

**Frontend** — `frontend/`
- Vite + React 18 + TypeScript (strict)
- Tailwind CSS
- cytoscape.js + cytoscape-dagre — graph visualization
- React Router v6 — two routes: `/` (dashboard) and `/graph`

## How to run

**Backend:**
```bash
cd backend
uv sync
uv run uvicorn app.main:app --reload --port 8000
```

**Frontend:**
```bash
cd frontend
npm install
npm run dev   # runs on port 5173, proxies /api → localhost:8000
```

Set `ANTHROPIC_API_KEY` in `backend/.env`. If absent or set to a dummy value, the app falls back to static scripted responses — the demo still works in full.

## The core idea

### Knowledge graph vs Context graph

A **knowledge graph** models *what is* — a static snapshot of entities and relationships. It answers "who is Alice, what is her goal, what exercises does she do."

A **context graph** adds two things a knowledge graph lacks:

1. **Time** — every action is a timestamped node, not a property update. The graph remembers that Alice skipped Monday *and then* reported knee pain *and then* skipped again. Order matters.
2. **Agent traces** — the agent's own decisions are written back into the graph as `DecisionTrace` nodes. The graph learns from what the agent did, not just what the client did.

### The two clocks

| Clock | What it answers | Where you see it |
|---|---|---|
| **State clock** | What is true *right now* | Client profile, Today's Program |
| **Event clock** | What happened, in what order | Training Trajectory, Context Graph |

The agent needs the *event clock* to decide. A snapshot that says "Alice has knee soreness" is not enough — you need to know it's the *third* soreness event in two weeks, that she skipped sessions in between, and that she has a prior tendinitis on record. That's a trajectory, not a state.

### How the agent decides

```
graph traversal → signals → deterministic rule → action
                                                     ↓
                                              LLM formats language
                                                     ↓
                                         DecisionTrace written to graph
```

The rule engine (`backend/app/agent/rules.py`) is fully deterministic:

| Condition | Action |
|---|---|
| adherence < 60% AND soreness ≥ 2 | DELOAD |
| avg soreness intensity ≥ 6 | DELOAD (override) |
| adherence < 75% OR soreness ≥ 1 | REDUCE |
| adherence ≥ 90% AND PR hit AND RPE ≤ 7 | PROGRESS |
| otherwise | MAINTAIN |

**The LLM never decides.** It only receives the decision and formats it into a readable message or adapted program. This is the key architectural constraint.

### The learning loop

Every agent run emits a `DecisionTrace` node into the graph, linked to the action nodes that caused it. Over time, the graph accumulates a history of *what the agent decided and why* — which itself becomes context for future decisions. This is what separates a context graph from a static knowledge base.

### The demo scenario

Alice Martin has:
- 2 weeks of training data (sessions, skips, soreness events)
- A prior left knee tendinitis on record
- A pattern of declining adherence + recurring knee and back soreness

When she types "I have knee pain today" in the chat:
1. The coach identifies the overtraining pattern from the graph (3 soreness events in 2 weeks, prior injury)
2. Recommends a rest day + adapted program
3. Clicking **Graph explained** shows exactly which nodes were consulted and why
4. Clicking **Accept program change** updates Today's Program to a rest day
5. Navigating to **/graph** shows the new soreness node added live to the context graph
6. Clicking **Reset** re-seeds the graph to its initial state

### Note on the chatbot

The chat responses are **scripted fallbacks** designed to illustrate the concept for the demo. With a real `ANTHROPIC_API_KEY`, the responses are generated by Claude using the actual graph data (soreness history, injury notes, current signals) as context. The architecture is the same either way — the LLM only speaks, the graph decides.

## Graph schema

```
Client ──HAS_GOAL──▶ Goal
Client ──FOLLOWS──▶ Program
Client ──HAS_ACTION──▶ SessionLogged     (load_kg, RPE, ts)
Client ──HAS_ACTION──▶ SessionSkipped    (reason, ts)
Client ──HAS_ACTION──▶ SorenessReported  (zone, intensity, ts)
Client ──HAS_ACTION──▶ PRHit             (exercise, weight, ts)
Client ──HAS_DECISION──▶ DecisionTrace   (action, rule, inputs, ts)
SorenessReported ──CAUSED──▶ DecisionTrace
SessionSkipped   ──CAUSED──▶ DecisionTrace
SessionSkipped   ──PREDICTS p=0.7──▶ WeekSkipped
SorenessReported ──TRIGGERS p=0.8──▶ AgentDeload
```

Probabilistic edges (`p < 1`) are shown as dashed lines in the graph view.
