# Personal Trainer — Context Graph Demo

> Technical interview project — built to demonstrate understanding of a context graph architecture for an AI coaching application.

## The idea in one sentence

An agent makes decisions by traversing a **timestamped graph of actions**. The LLM only formats language — it never decides.

---

## Knowledge graph vs Context graph

Most AI apps wire an LLM to a knowledge graph: a static snapshot of who the user is and what they have. That answers "what program is Alice following?" It cannot answer "should Alice deload this week?"

A **context graph** adds two things a knowledge graph lacks:

**1. Time.** Actions are first-class timestamped nodes — not property updates that overwrite history. The graph remembers that Alice skipped Monday, reported knee pain Tuesday, then skipped again Thursday. The *sequence* is the signal.

**2. Agent traces.** The agent writes its own decisions back into the graph as `DecisionTrace` nodes. The graph accumulates a history of what was decided, why, and based on what inputs — and that history becomes context for future decisions.

### The two clocks

| Clock | What it captures | Where you see it in the app |
|---|---|---|
| **State clock** | What is true *right now* | Client profile, Today's Program |
| **Event clock** | What happened, in what order | Training Trajectory, Context Graph |

A snapshot says Alice has knee soreness. A trajectory says it's the third soreness event in two weeks, she has a prior tendinitis on record, and her adherence has dropped 30% — that's an overtraining signal, not a one-off. The agent needs the event clock to see the pattern.

## Architecture

```
frontend/          Vite + React + TypeScript + Tailwind
  /                Dashboard — state clock (profile, program) + event clock (trajectory) + virtual coach
  /graph           Live context graph visualization (cytoscape.js + dagre layout)

backend/           FastAPI — pure JSON API
  graph/
    model.py       networkx DiGraph — the context graph (singleton, in-memory)
    seed.py        Synthetic data: Alice (overtraining pattern) and Bob (thriving)
    traversal.py   Computes signals by graph traversal — no FastAPI dependency
  agent/
    rules.py       Deterministic rule engine → DELOAD / REDUCE / MAINTAIN / PROGRESS
    llm.py         Anthropic SDK — language only, never decisions
    decision.py    Orchestrates traversal → rule → LLM → DecisionTrace
  api/             FastAPI routes + Pydantic schemas
```

### How the agent works

```
Client node
    └── HAS_ACTION ──▶ [SessionLogged, SessionSkipped, SorenessReported, ...]
                                │
                      graph traversal (traversal.py)
                                │
                             signals
                      (adherence_14d, soreness_count_7d,
                       avg_rpe_7d, drop_off_probability ...)
                                │
                      deterministic rule (rules.py)
                                │
                             action
                      DELOAD / REDUCE / MAINTAIN / PROGRESS
                                │
                      LLM formats language (llm.py)       ← only step involving the LLM
                                │
                      DecisionTrace node written to graph  ← learning loop
```

**The LLM never sees the decision problem.** It receives the already-made decision and formats it into a readable message or adapted program. This is the key constraint the architecture enforces.

## Demo scenario

The demo follows **Alice Martin**, a client with:
- 2 weeks of training history (sessions, skips, soreness events)
- A prior left knee tendinitis on record
- Declining adherence + recurring knee/back soreness → overtraining pattern

**Flow:**
1. Alice types "I have knee pain today" in the virtual coach chat
2. The coach identifies the pattern from the graph: 3 soreness events in 2 weeks + prior injury record
3. **Graph explained** reveals exactly which nodes were consulted and why
4. **Accept program change** → Today's Program updates to a rest day
5. Navigate to **/graph** → the new soreness node appears live in the graph
6. **Reset** → graph re-seeds to initial state, chat clears, program restores

> **Note on the chatbot:** Chat responses are scripted fallbacks, designed to reliably illustrate the concept during the demo without requiring a live API call. With a real `ANTHROPIC_API_KEY`, responses are generated by `claude-sonnet-4-6` using actual graph data (injury history, soreness events, signals) injected as context. The architecture is identical either way — the LLM only speaks, the graph decides.

**Backend:**

```bash
cd backend

uv run uvicorn app.main:app --reload --port 8000
```

**Frontend:**
```bash
cd frontend
npm install
npm run dev   # http://localhost:5173 — proxies /api → localhost:8000
```

```
ANTHROPIC_API_KEY=your-key-here
```

## Graph schema

```
Client ──HAS_GOAL──────▶ Goal
Client ──FOLLOWS───────▶ Program
Client ──HAS_ACTION────▶ SessionLogged      { load_kg, RPE, ts }
Client ──HAS_ACTION────▶ SessionSkipped     { reason, ts }
Client ──HAS_ACTION────▶ SorenessReported   { zone, intensity, ts }
Client ──HAS_ACTION────▶ PRHit              { exercise, weight, ts }
Client ──HAS_DECISION──▶ DecisionTrace      { action, rule_evaluated, inputs, ts }

SorenessReported ──CAUSED──────────▶ DecisionTrace
SessionSkipped   ──CAUSED──────────▶ DecisionTrace
SessionSkipped   ──PREDICTS p=0.7──▶ WeekSkipped
SorenessReported ──TRIGGERS p=0.8──▶ AgentDeload
```

Dashed edges in the graph view = probabilistic (p < 1).
Diamond nodes = `DecisionTrace` — the agent's decisions, written back into the graph.
